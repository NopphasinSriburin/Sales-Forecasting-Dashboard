# โปรเจกต์นี้ทำงานยังไง — อธิบายสำหรับทำความเข้าใจ

เอกสารนี้อธิบายว่าโปรเจกต์ทำงานยังไงตั้งแต่ต้นจนจบ เขียนให้เข้าใจง่าย เหมาะสำหรับทบทวนก่อนสัมภาษณ์

---

## 1. ภาพรวม: โปรเจกต์นี้ทำอะไร

รับข้อมูลยอดขายในอดีต แล้วพยากรณ์ว่ายอดขายในอนาคตจะเป็นเท่าไหร่ พร้อมแสดงผลเป็น dashboard ที่ผู้บริหารดูแล้วเข้าใจทันที

**Flow หลัก:**
```
ข้อมูลยอดขาย → เตรียมข้อมูล → เทรนโมเดล → พยากรณ์ → แสดงผลบน dashboard
```

---

## 2. โครงสร้างไฟล์ และหน้าที่ของแต่ละไฟล์

โปรเจกต์แยกเป็น 3 ส่วนหลัก (separation of concerns — แต่ละไฟล์ทำหน้าที่เดียว):

### `app.py` — หน้าจอและการควบคุม (UI layer)
เป็นไฟล์ที่ผู้ใช้เห็น ทำหน้าที่:
- รับ input จากผู้ใช้ (เลือกโหมด, อัปโหลดไฟล์, เลือกโมเดล)
- ตรวจสอบข้อมูล (validation) และแจ้งเตือน
- เรียกใช้โมเดลจาก `forecasting.py`
- วาดกราฟ, KPI, ตาราง

### `src/forecasting.py` — สมองของระบบ (logic layer)
เป็นหัวใจที่ทำการพยากรณ์จริง ไม่เกี่ยวกับหน้าจอเลย ทำหน้าที่:
- เทรนโมเดล Prophet และ XGBoost
- สร้าง features สำหรับ XGBoost
- วัดความแม่นยำ
- ทำ backtesting

### `src/ui.py` — หน้าตาและสไตล์ (presentation layer)
- เก็บ icon (Lucide SVG) และ CSS theme
- แยกออกมาเพื่อให้ `app.py` สะอาด ไม่ปนโค้ดแต่งหน้าตา

**ทำไมต้องแยกไฟล์?** ถ้าใครถามตอนสัมภาษณ์: การแยก UI / logic / style ออกจากกันทำให้แก้โค้ดง่าย ทดสอบ logic ได้โดยไม่ต้องเปิดหน้าจอ และถ้าจะเปลี่ยนจาก Streamlit เป็นอย่างอื่น ก็แก้แค่ `app.py` โดยไม่แตะ logic

## 3. ขั้นตอนการทำงานละเอียด (เดินตาม flow)

### ขั้นที่ 1: รับข้อมูล
ผู้ใช้เลือกได้ 2 ทาง:
- **โหมด Demo** — ใช้ไฟล์ `data/sales_raw.csv` ที่เตรียมไว้ (ข้อมูลสังเคราะห์ 3 ปี)
- **โหมด Upload** — อัปโหลด CSV เอง แล้วระบบให้จับคู่ว่าคอลัมน์ไหนคือวันที่/ยอดขาย/หมวด

### ขั้นที่ 2: ตรวจสอบข้อมูล (validation)
ก่อนเทรน ระบบเช็คว่า:
- คอลัมน์วันที่แปลงเป็นวันที่ได้จริงไหม (ถ้าเลือกผิดจะเตือน)
- คอลัมน์ยอดขายเป็นตัวเลขไหม (ถ้าเลือกเป็นข้อความจะเตือน)
- ข้อมูลมีอย่างน้อย 30 วันไหม

ถ้าผิดจะแสดงคำเตือนพร้อมวิธีแก้ แทนที่จะ error พัง

### ขั้นที่ 3: เตรียมข้อมูล
- แปลงวันที่ให้เป็นรูปแบบมาตรฐาน
- รวมยอดขายที่อยู่ในวันเดียวกัน (aggregate เป็นรายวัน)
- เรียงตามเวลา

### ขั้นที่ 4: เทรนโมเดล (หัวใจ)
นี่คือส่วนที่สำคัญที่สุด อธิบายละเอียดในหัวข้อ 4

### ขั้นที่ 5: แสดงผล
- KPI cards (ยอดคาดการณ์, ความแม่นยำ)
- กราฟยอดจริง vs ยอดพยากรณ์
- ตารางเทียบโมเดล, feature importance, backtesting
- ตารางสรุปรายเดือน + ปุ่มดาวน์โหลด

## 4. หัวใจ: การเทรนโมเดลทำงานยังไง

### 4.1 Time-based Split (จุดที่สำคัญที่สุดของ time series)

**ปัญหา:** ในงานทั่วไป เราแบ่งข้อมูล train/test แบบสุ่ม (random) แต่ในงาน time series **ห้ามทำแบบนั้นเด็ดขาด**

**เพราะอะไร:** ถ้าสุ่ม โมเดลอาจได้เห็นข้อมูล "อนาคต" ตอนเทรน แล้วเอามาทาย"อดีต" ซึ่งไม่สมจริง (เรียกว่า data leakage) เหมือนดูเฉลยก่อนสอบ

**วิธีที่ถูก:** แบ่งตามเวลา — เอาข้อมูลช่วงแรกเป็น train ช่วงท้ายเป็น test
```
|-------- train (85%) --------|--- test (15%) ---|
2023 -------------------- 2025 ก.ย.      2025 ธ.ค.
```
แบบนี้เหมือนสถานการณ์จริง: ใช้อดีตทำนายอนาคต

### 4.2 โมเดลที่ 1: Prophet

- พัฒนาโดย Meta (Facebook) ออกแบบมาสำหรับ time series โดยเฉพาะ
- เก่งเรื่องจับ **ฤดูกาล** (seasonality) และ **วันหยุด** อัตโนมัติ
- ใช้ง่าย ไม่ต้องปรับแต่งเยอะ เหมาะกับข้อมูลที่มี pattern ชัด
- ในโปรเจกต์นี้เราใส่ข้อมูลวันหยุด/โปรโมชันเข้าไปช่วยด้วย

### 4.3 โมเดลที่ 2: XGBoost

XGBoost เป็นโมเดล Machine Learning ทั่วไป ไม่ได้ออกแบบมาเพื่อ time series โดยตรง เราจึงต้อง **สร้าง features เอง** ให้มันเข้าใจเวลา:

**Lag features** — ยอดขายย้อนหลัง
- `lag_1` = ยอดขายเมื่อวาน
- `lag_7` = ยอดขาย 7 วันก่อน (สัปดาห์ก่อน)
- `lag_14`, `lag_30` = 2 สัปดาห์ / 1 เดือนก่อน

แนวคิด: ยอดขายวันนี้มักสัมพันธ์กับยอดขายวันก่อนๆ

**Rolling features** — ค่าเฉลี่ย/ความผันผวนย้อนหลัง
- `roll_mean_7` = ค่าเฉลี่ยยอดขาย 7 วันล่าสุด
- `roll_std_7` = ความผันผวน 7 วันล่าสุด

แนวคิด: บอกแนวโน้มระยะสั้น

**Calendar features** — ข้อมูลปฏิทิน
- วันในสัปดาห์, เดือน, เป็นวันหยุดสุดสัปดาห์ไหม

แนวคิด: ยอดขายเสาร์-อาทิตย์อาจต่างจากวันธรรมดา

### 4.4 Recursive Forecasting (จุดที่ XGBoost ต่างจาก Prophet)

**ปัญหา:** XGBoost ต้องใช้ "ยอดขายเมื่อวาน" มาทำนายวันนี้ แต่ถ้าจะทำนายอนาคต 90 วัน เราไม่มี "ยอดขายเมื่อวาน" ของวันในอนาคต

**วิธีแก้ — ทำนายทีละวันแบบต่อเนื่อง:**
```
ทำนายวันที่ 1 → เอาผลที่ได้เป็น "เมื่อวาน" ของวันที่ 2
ทำนายวันที่ 2 → เอาผลที่ได้เป็น "เมื่อวาน" ของวันที่ 3
... ทำไปเรื่อยๆ จนครบ 90 วัน
```
เรียกว่า recursive forecasting

### 4.5 วัดความแม่นยำด้วยอะไร

- **MAPE** (Mean Absolute Percentage Error) — คลาดเคลื่อนกี่ % โดยเฉลี่ย (ผู้บริหารเข้าใจง่ายสุด: "พลาด 8%")
- **RMSE** — ค่าคลาดเคลื่อนเป็นหน่วยเงิน เน้นโทษความผิดพลาดใหญ่ๆ
- **MAE** — ค่าคลาดเคลื่อนเฉลี่ยเป็นหน่วยเงิน

เกณฑ์ธุรกิจทั่วไป: MAPE < 15% ถือว่าใช้ได้

## 5. Backtesting — ทำไมวัดผลรอบเดียวไม่พอ

**ปัญหา:** ถ้าวัดความแม่นแค่ช่วงเดียว อาจฟลุ้ค (บังเอิญช่วงนั้นทำนายง่าย)

**วิธีแก้ — เลื่อนหน้าต่างทดสอบหลายรอบ (rolling-window):**
```
รอบ 1: เทรนถึง ก.ค. 2024 → ทดสอบ ส.ค. 2024
รอบ 2: เทรนถึง ธ.ค. 2024 → ทดสอบ ม.ค. 2025
รอบ 3: เทรนถึง มิ.ย. 2025 → ทดสอบ ก.ค. 2025
รอบ 4: เทรนถึง ธ.ค. 2025 → ทดสอบ ม.ค. 2026
```
แล้วดูว่า MAPE ทุกรอบใกล้เคียงกันไหม ถ้าใกล้ = โมเดลน่าเชื่อถือจริง ไม่ใช่ฟลุ้ค

## 6. Performance — ทำให้เร็วขึ้นยังไง

**ปัญหา:** Streamlit จะรันโค้ดใหม่ทั้งหมดทุกครั้งที่ผู้ใช้ขยับอะไรบนหน้าจอ (เช่นเลื่อน slider) ทำให้เทรนโมเดลใหม่ซ้ำๆ ทั้งที่ข้อมูลเดิม

**วิธีแก้ 2 อย่าง:**

1. **Cache (`@st.cache_data`)** — จำผลการเทรนไว้ ถ้า input เหมือนเดิม (ข้อมูล + โมเดลเดิม) ดึงผลเก่ามาใช้เลย ไม่เทรนใหม่

2. **เทรน horizon คงที่แล้วตัดเอา** — เทรนพยากรณ์ 180 วันไว้ตายตัว ถ้าผู้ใช้เลื่อน slider ดู 30/90/180 วัน ก็แค่ตัดผลที่มีอยู่แล้ว ไม่ต้องเทรนใหม่

ผลลัพธ์: หลังเทรนครั้งแรก การเปลี่ยน filter/slider จะเร็วแทบทันที

## 7. คำถามสัมภาษณ์ที่อาจเจอ + แนวตอบ

**Q: ทำไมเลือก Prophet กับ XGBoost?**
A: Prophet เป็น statistical model ที่เก่งเรื่องฤดูกาลและใช้ง่าย ส่วน XGBoost เป็น ML ที่ยืดหยุ่นกว่าถ้ามี feature ดีๆ ผมใส่ทั้งคู่เพื่อเทียบว่าแบบไหนเหมาะกับข้อมูลชุดนี้ — เป็นการแสดงว่าไม่ยึดติดโมเดลเดียว

**Q: ทำไมห้าม random split ใน time series?**
A: เพราะจะเกิด data leakage — โมเดลเห็นอนาคตตอนเทรนแล้วเอามาทายอดีต ไม่สมจริง ต้องแบ่งตามเวลาเท่านั้น

**Q: feature ไหนสำคัญสุด แล้วบอกอะไรเรา?**
A: lag_7 (ยอดขาย 7 วันก่อน) สำคัญสุด แปลว่ายอดขายมี pattern รายสัปดาห์ชัด — วันเดียวกันของสัปดาห์ก่อนทำนายได้ดี

**Q: ถ้าข้อมูลจริงพยากรณ์ไม่แม่นล่ะ?**
A: เจอมาแล้วกับ dataset จริง (Superstore) ที่ MAPE สูงเพราะข้อมูลผันผวนมาก บทเรียนคือไม่ใช่ทุกข้อมูลพยากรณ์ได้แม่น การรู้ว่าเมื่อไหร่ข้อมูลไม่เหมาะก็เป็นทักษะ วิธีรับมือคือ aggregate เป็นรายสัปดาห์/เดือนเพื่อลด noise

**Q: จัดการ error ยังไง?**
A: ใส่ validation ก่อนเทรน (เช็คคอลัมน์, เช็คจำนวนข้อมูล) และ catch error ให้แสดงเป็นคำเตือนที่ผู้ใช้เข้าใจ แทนที่จะพังทั้งแอป

## 8. สรุปสั้นๆ ว่าโปรเจกต์นี้เก่งตรงไหน

1. ครบ end-to-end — ตั้งแต่ข้อมูลดิบถึง dashboard ใช้งานจริง
2. ทำถูกหลัก time series — time-based split, backtesting
3. ใช้งานได้จริง — อัปโหลดข้อมูลเอง, validation, จัดการ error
4. คิดเรื่อง performance — caching
5. เทียบโมเดลอย่างมีเหตุผล — ไม่ได้ใช้โมเดลเดียวแบบมั่วๆ
